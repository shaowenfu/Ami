# 2026-05-27 Ami 上下文编排重构日记

今天这轮工作从一个很关键的产品判断开始：Ami 的智能和情商，不应该主要依赖一段聪明的 prompt，而应该发生在上下文工程，尤其是数据库查询和上下文编排阶段。

## 对话回顾

一开始，我们先梳理了当前后端智能体设计现状。旧实现已经具备消息持久化、SSE 流式回复、LLM 调用和 Mem0 记忆读写，但核心问题是上下文模型太薄：`ChatContextBuilder` 只是把最近消息和记忆片段拼进 prompt，`privacy_rule` 被放在提示词里承担过多防御责任，而 `AgentProfile.tone` 也不足以表达 Ami 在关系中的连续自我认知。

随后，产品理念被重新校准：Ami 不是一个普通客服式机器人，也不是两个人之间的裁判。它更像两人关系中的共同好友、长期在场者和关系记忆 keeper。于是，核心协议必须承认三类信息源是相通的：用户 A 与 Ami 的私聊、用户 B 与 Ami 的私聊、两位用户与 Ami 的三人群聊。区别不在于 Ami 是否“知道”，而在于当前场景下如何使用、如何表达、哪些只能作为背景理解。

在这个基础上，我们确定了几个设计约束：

- Agent 需要两套上下文拼接机制：单人私聊与三人群聊。
- `AgentProfile` 保持极简，只保留 `name`、`self_recognition`、`prompt`。
- 静态身份、使命、关系位置和边界不进数据库，统一硬编码到基础 prompt。
- `Space` 增加三个简单字符串：`user_a_profile`、`user_b_profile`、`relationship_summary`。
- 睡眠机制负责每天消化前一天信息，更新长期摘要和 Ami 自我认知。
- 心跳机制是主动发消息能力的基石，但首版必须保守，先做判断链路，不贸然发送。

## 执行过程

第一步是把完整方案落盘到 `docs/tasks/context-orchestration-core-protocol-plan.md`。这份方案明确了核心上下文协议、私聊与群聊 prompt 结构、数据模型变化、查询策略、睡眠机制和心跳机制。

第二步进入代码实现。后端模型中删除了旧的 `tone`，将 `AgentProfile` 重构为三个字段；`DBSpace` 与响应模型加入两个用户画像和关系摘要。空间创建、种子脚本、前端 API 类型和空间列表展示也同步调整。

第三步是核心：用 `ContextOrchestrator` 替代旧的 `ChatContextBuilder`。新的编排器不再只返回一个“上下文块”，而是按私聊和三人群聊分别构造完整 System Prompt。私聊模式会拉取当前用户私聊近期消息、共享群聊近期消息、当前私聊和共享长期记忆，并注入长期画像与关系摘要；群聊模式只拉取共享群聊与共享记忆，同时使用长期摘要作为低敏背景。另一方私聊原文不会进入 prompt。

第四步是接入消息生成链路。`MessageService.generate_agent_reply()` 现在会先重新加载当前 `Space`，再把 space、当前用户、room scope 和 query 交给 `ContextOrchestrator`，由它产出完整 System Prompt。旧的 `_build_system_prompt()` 和 prompt 级 `privacy_rule` 被删除。

第五步补上睡眠与心跳的服务骨架。`SleepDigestService` 可以读取前一天三类房间消息，并在非 dry-run 模式下调用 LLM 生成新的用户画像、关系摘要和 Ami 自我认知。`HeartbeatService` 目前只做共享群聊空闲判断，不实际发送消息，并且对没有历史消息的新空间保持沉默。

最后做了一轮代码质量打磨。删除了 `RenderedContext` 这层不必要包装，让 `ContextOrchestrator.build_context()` 直接返回字符串；压缩并打磨了核心提示词；收紧了睡眠服务里的私聊消息读取逻辑；修正了可变默认值；让心跳策略更克制。

## 当前结果

现在后端已经形成一个更清晰的上下文协议：

- 数据库查询阶段先做强边界，避免不该进入当前上下文的原始数据成为候选。
- Prompt 中不再试图用死规则替代安全边界，而是解释来源语义，让 Agent 做关系表达判断。
- Ami 的自我认知、用户画像和关系摘要成为每轮回复的稳定上下文。
- 私聊与三人群聊有明确不同的上下文结构。
- 睡眠和心跳已经有服务边界，后续可以继续接调度器和主动发送策略。

验证方面，后端源码通过 `python -m py_compile`，前端通过 `npx tsc --noEmit`。旧的 `ChatContextBuilder`、`agent_profile.tone`、`RenderedContext` 等核心残留已经清理。

## 体感

这次重构真正重要的部分，不是多写了几个字段或换了一个类名，而是把 Ami 的“知道什么”和“怎么说出来”拆开了。这个拆分让系统更像一个可信任的共同好友，而不是一个靠 prompt 约束秘密泄漏的聊天机器人。

接下来最值得继续推进的是两件事：一是让睡眠机制真正跑起来，并设计好摘要质量评估；二是给心跳机制加上更细的触发条件和发送前审查，保证 Ami 的主动性是体贴的，而不是打扰的。
