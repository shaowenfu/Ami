交互架构设计：Ami 核心链路与界面层级

Interaction Architecture: Core Workflows and UI Hierarchy for Ami

1. 概述 (Overview)

本文档定义了 Ami 双人 Agent 应用的顶层交互架构。Ami 采用独特的“全局-空间”双层级设计，在物理和感知上严格隔离了“个体身份”与“关系实体”，从而在保证极高隐私安全感的同时，营造出极具沉浸感的“双人结界”体验。

2. 界面层级拆解 (UI Hierarchy)

整个应用被划分为两个平行的界面空间，通过特定的操作（如进入/退出空间）进行切换，两个层级的底部导航栏（Bottom Tab Bar）互不连通。

2.1 全局层 (Global Layer: Individual Identity)

此层级以单个用户（User Account）为中心。用户在这里管理个人信息，并总览所有的关系纽带。

导航结构: 无全局底部导航栏（或仅在必要时展示简单的全域导航）。

核心页面:

认证模块 (Auth): 手机号验证码一键登录、基础资料初始化。

空间概览页 (Space Dashboard - 默认首页):

主视觉: 卡片列表流，展示已建立连接的“关系空间”（如：“我与伴侣”、“父子连线”）。

状态入口: 顶部的“邀请信箱”，展示待处理的关系绑定请求。

行动点 (CTA): 悬浮按钮 (FAB) 用于唤起“创建新关系”流程。

全局个人主页 (Global Profile):

账号安全（手机号绑定等）。

全局隐私与通知设定。

关于应用、帮助反馈。

2.2 空间层 (Space Layer: Relationship Entity)

用户在概览页点击任一关系卡片后，进入该层级。此层级呈现的是一个完整的“App in App”，所有的交互均以当前被点击的这段亲密关系以及专属的 Ami Agent 为上下文。

导航结构: 专属的底部导航栏 (Bottom Tab Bar)，仅在当前空间内有效。

核心页面:

聊天室 (Chat - 默认首页):

顶部切换器 (Top Tab Bar): 核心防误触设计。

Tab A - 私聊 (Private): 仅限 [当前用户 + Ami] 参与，用于倾诉、咨询和信息收集。

Tab B - 群聊 (Shared): [用户 A + 用户 B + Ami] 参与，用于关系互动和共同维护。

关系工具箱 (Tools):

当前预留为空白画布。未来承载纪念日倒数、情感周报、共同打卡等扩展功能。

空间专属主页 (Space Profile):

仅控制当前关系的设定：如共同给 Ami 命名/设定性格。

退出路径: 提供明确的 < 返回概览> 操作，使用户退出空间层，回到全局层。

3. 核心交互链路：关系缔结 (Core Workflow: Relationship Binding)

关系的建立是应用的核心转化漏斗，设计原则为“精确搜索、双向确认”。

发起方 (Initiator): 在全局层触发新建动作 -> 输入对方手机号精确搜索 -> 发送附带留言的邀请请求。

状态同步: 发起方界面产生“等待对方确认”的占位卡片；接收方触发系统级推送（Push Notification）。

接收方 (Receiver): 进入全局层的邀请信箱 -> 查看留言 -> 点击确认同意。

建立完成: 双方全局列表同时点亮新的关系卡片，后台触发该关系专属 Ami 的初始化操作。

4. 设计原则总结 (Design Principles)

空间隔离感 (Spatial Segregation): “全局”到“空间”的切换必须有明显的视觉隐喻（如全屏转场动画），强化“进入专属领地”的认知。

隐私防呆 (Privacy Error-Proofing): 空间层的“私聊”与“群聊”在视觉主题上必须有强烈的区分度（如背景色、Ami 头像样式的微妙变化），以杜绝用户在私密吐槽时发错窗口导致的社交事故。
